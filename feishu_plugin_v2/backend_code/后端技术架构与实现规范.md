# 后端技术架构与实现规范

## 1. 技术栈选型
* **语言**: Python 3.10+
* **Web 框架**: Flask (轻量级，易于快速开发)
* **飞书 SDK**: `baseopensdk` (官方提供的服务端 SDK)
* **包管理**: Poetry (依赖管理与环境隔离)
* **部署环境**: Docker / K8s / 云服务器 (需支持 HTTPS)

## 2. 目录结构规范
基于 `BaseOpenSDK-Python-Playground` 扩展，采用标准的 **Service-Controller** 分层架构：

```text
backend/
├── main.py                # 应用入口 (Flask App 初始化, CORS 配置)
├── .env                   # [敏感] 环境变量 (AppID, Secret)
├── pyproject.toml         # 依赖定义
├── services/              # 核心业务逻辑层 (Service Layer)
│   ├── __init__.py
│   ├── lark_service.py    # 飞书 API 封装 (Token, Drive, Bitable CRUD)
│   └── sign_service.py    # 签字业务逻辑 (会签/或签仲裁, 状态流转)
├── routes/                # 路由层 (Controller Layer)
│   ├── __init__.py
│   └── api_routes.py      # 定义 API 接口，参数校验，调用 Service
├── templates/             # 模版文件
│   └── sign.html          # 签字 H5 页面 (Jinja2 模板)
└── utils/                 # 工具类
    └── qrcode_utils.py    # 二维码生成工具
```

## 3. 核心开发规范

### 3.1 字段标识 (Field ID vs Name)
> [!IMPORTANT]
> **严禁使用字段名称 (Field Name) 进行数据更新！**

*   **原因**：用户可能会修改多维表格的列名（如将“签字状态”改为“状态”），导致代码失效。
*   **规范**：
    1.  在程序启动或操作前，先调用 `get_bitable_fields` 获取 `Field Name -> Field ID` 的映射表。
    2.  所有 `update_record` 操作必须使用 **Field ID** 作为 Key。

### 3.2 异常处理
*   **全局捕获**：在 `api_routes` 层捕获所有未处理异常，统一返回 JSON 格式错误。
*   **飞书 API 错误**：对于 `lark_service` 抛出的 API 异常（如 Token 失效），应尝试自动刷新 Token 并重试一次，若仍失败则向上抛出。

### 3.3 接口响应标准
所有 API 必须遵循统一的 JSON 响应格式：
```json
{
  "code": 0,          // 0: 成功, 非0: 失败
  "msg": "success",   // 成功提示或错误描述
  "data": { ... }     // 业务数据
}
```

### 3.4 安全性
*   **CORS**：仅允许受信任的前端域名跨域访问。
*   **参数校验**：所有接收的参数（尤其是 `record_id`, `table_id`）必须进行格式校验，防止注入攻击。
