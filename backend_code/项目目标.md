# 后端项目目标 (BaseOpenSDK-Python-Playground)

## 1. 项目定位
本项目是飞书多维表格电子签名系统的**核心业务中枢**。作为 "Backend-Centric" 架构的核心，它全权负责与飞书 OpenAPI 的交互、数据的增删改查以及复杂业务逻辑的实现。

## 2. 核心职责

### 2.1 业务逻辑处理 (Business Logic)
*   **签字模式仲裁**：
    *   **或签 (OR)**：接收到签名后，直接覆盖原有附件，更新状态为“已签字”。
    *   **会签 (AND)**：接收到签名后，追加到附件列表。计算（当前签名数 / 设定总人数），若未达标则更新进度文本（如“签字中 1/3”），若达标则更新状态为“已签字”。
*   **历史签名管理**：
    *   维护 `User_ID` -> `File_Token` 的映射关系。
    *   提供查询接口，判断用户是否已有历史签名。
    *   提供复用接口，直接使用历史 Token 完成签署。

### 2.2 数据与权限管理 (Data Authority)
*   **Token 管理**：负责 `tenant_access_token` 的获取与自动刷新，确保 API 调用不中断。
*   **列自动化**：
    *   根据配置自动在多维表格中创建“签字确认”、“签字状态”、“签字附件”等列。
    *   **动态二维码**：根据请求参数动态判断是否需要创建或删除“签字二维码”列。
*   **安全写入**：所有对多维表格的写入操作（Update Record）均由后端执行，确保数据格式正确且符合业务规则。

### 2.3 API 服务 (API Provider)
*   **H5 页面托管**：提供 `/sign` 路由渲染 HTML 页面。
*   **RESTful API**：
    *   `POST /api/batch/generate-links`：批量生成签字链接与二维码。
    *   `GET /api/sign/info`：获取签字页初始化信息（行数据、历史签名状态）。
    *   `POST /api/sign/submit`：处理签名提交（Base64 图片上传或 Token 复用）。

## 3. 关键技术指标
*   **并发处理**：支持多用户同时访问签字页，确保 SQLite 或文件操作无锁冲突（若使用轻量级存储）。
*   **响应速度**：签字提交接口 (`/submit`) 应在 3 秒内完成（含图片上传至飞书 Drive 的时间）。
*   **容错性**：当飞书 API 报错（如 Token 过期、限流）时，应有自动重试机制或友好的错误提示。

## 4. 目录结构说明
*   `main.py`: Flask 应用入口与 CORS 配置。
*   `services/lark_service.py`: 飞书 OpenAPI 封装（底层）。
*   `services/sign_service.py`: 签字业务逻辑实现（核心）。
*   `routes/api_routes.py`: 前后端交互接口定义。
*   `templates/sign.html`: H5 签字页面模板。
