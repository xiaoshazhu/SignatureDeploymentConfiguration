# 项目目标与需求定义

## 1. 项目概述
本项目旨在开发一个基于飞书多维表格的电子签名插件系统。项目采用**后端为中心 (Backend-Centric)** 的架构，前端负责交互与触发，后端负责核心业务逻辑与数据写入，以确保数据的一致性与安全性。

## 2. 核心架构与分工

### 2.1 架构模式：Backend-Centric
*   **前端 (Vue)**：仅作为“触发器”和“展示层”。不直接修改多维表格数据，而是调用后端 API 执行操作。
*   **后端 (Python)**：作为“执行器”和“数据中心”。拥有多维表格的写入权限，处理所有业务逻辑（如生成链接、更新状态、上传附件）。

### 2.2 角色分工

#### A. 多维表格插件 (前端 Vue)
**定位**：配置面板与操作触发入口。
*   **配置管理**：设置签字模式（会签/或签）、签字人数、二维码开关等。
*   **操作触发**：
    *   **批量生成**：调用后端 API 批量生成全量数据的签字链接。
    *   **单行生成**：针对单行数据，自定义配置并生成链接。
*   **状态展示**：显示操作日志与结果统计。

#### B. 签字 H5 页面 (前端 Vue/HTML)
**定位**：移动端/PC 端通用的签字终端。
*   **信息展示**：通过手风琴折叠面板展示当前行详细数据，防止误签。
*   **签字功能**：提供 Canvas 画板进行手写签名。
*   **历史复用**：自动检索当前用户的历史签名，支持一键复用。

#### C. 后端服务 (Python)
**定位**：业务中枢，全权负责数据流转。
*   **链接生成**：生成带签名保护或唯一标识的签字 URL。
*   **自动化列管理**：
    *   自动维护“签字确认”（超链接）、“签字状态”（单选）、“签字附件”（附件）、“签字二维码”（附件）等列。
    *   **动态二维码列**：根据配置开关，自动创建或删除“签字二维码”列。
*   **状态流转管理**：
    *   **签字确认列**：初始("请点击签字") -> 进行中("签字中 1/3") -> 完成("已签字")。
    *   **签字状态列**：初始("未签字") -> 完成("已签字")。
*   **核心业务逻辑**：
    *   **会签 (And-Sign)**：追加签名，所有人签完才更新状态为完成。
    *   **或签 (Or-Sign)**：覆盖签名，任意一人签完即更新状态为完成。
    *   **防重复提交**：校验当前状态，避免已完成的单据被重复签署。

## 3. 详细功能特性

### 3.1 签字配置
*   **全局配置**：通过插件面板设置默认的签字模式和人数。
*   **单行配置**：允许对特定行单独设置签字模式（如某行需要 5 人会签），配置参数加密或编码在 URL 中。

### 3.2 签字流程
1.  **用户进入**：点击多维表格链接或扫描二维码进入 H5。
2.  **身份核对**：查看手风琴面板中的详细信息。
3.  **签名动作**：
    *   **手写**：在画板书写并提交。
    *   **复用**：点击“复用历史签名”（若有）。
4.  **提交处理**：后端接收签名图，上传至飞书 Drive，更新多维表格附件列和状态列。

### 3.3 历史签名 (History Signature)
*   **存储**：后端维护用户 ID 与签名文件 Token 的映射关系（可存储在独立表或云文档）。
*   **复用**：用户再次签字时，前端查询后端接口，若存在历史签名则显示复用按钮。
